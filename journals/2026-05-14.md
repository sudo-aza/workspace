## latex-6: Fix Chapter Opener Pages

VM was reset. Re-cloned all 3 repos (workspace, todo-list, latex-template). Re-installed TeXLive from scratch (added tikzfill, colortbl, multirow to install script).

### Problem
The chapter divider pages (`\chapterpage`) were completely broken. VLM confirmed: "almost entirely blank" with only "a small dark navy blue rectangle in the top-left corner — no visible text." The dark background fill rendered, but all text nodes (CHAPTER label, title, decorative rules) were invisible. This affected all 3 chapter pages in the sample document.

### Root cause
The original implementation used `\begin{tikzpicture}[remember picture, overlay]` with `current page` node references for all text placement. This approach requires two compilation passes, and even after two passes, the text nodes failed to appear. The `remember picture` mechanism appears unreliable in LuaLaTeX + TeXLive 2026 for full-page overlays.

### Solution: newgeometry approach
Replaced the `remember picture, overlay` design with a `\newgeometry` + regular tikzpicture approach:

1. `\newgeometry{paper=a4paper, top=0pt, bottom=0pt, left=0pt, right=0pt}` — removes all margins
2. `\begin{tikzpicture}` with `\useasboundingbox (0,0) rectangle (\paperwidth, \paperheight)` — fills exact page size
3. All elements placed with absolute coordinates relative to the tikzpicture origin — no `remember picture` needed
4. `\clearpage` + `\restoregeometry` — returns to normal margins for content

### New design
- **Top 40% dark band** (titleDark) — clean header area
- **Left accent stripe** (titleRule, 5pt) — subtle visual anchor
- **Thin accent line** at band boundary (titleAccent)
- **Large watermark number** (140pt, 7% opacity) — subtle "1" in the dark area
- **"CHAPTER N" label** — gray "CHAPTER" + lighter number
- **Chapter title** — 30pt bold white text
- **Decorative rule** under title

### VLM Verification
- Chapter 1: **8/10** — "text is clear and legible against the dark header background"
- Chapter 2: **8/10** — "'CHAPTER 2' and 'Consensus Algorithms' text are visible"
- Content pages: still render correctly, no regressions

### Files changed
- `latex-template/sudodoc.sty` — v1.0.0 → v1.1.0, rewrote `\chapterpage`
- `latex-template/sample.pdf` — recompiled (9 pages)
- `latex-template/output/page-*.png` — updated previews
- `workspace/scripts/install-texlive.sh` — added tikzfill, colortbl, multirow

### Lessons
- `remember picture, overlay` is unreliable for full-page text in LuaLaTeX TeXLive 2026 — avoid it for critical content
- `\newgeometry` + regular tikzpicture is more robust for full-bleed pages
- Always verify with VLM after compilation — "compiles without errors" does not mean text is visible
- When editing todo.md, use Write instead of Edit for large structural changes to avoid formatting corruption

## latex-7: Fix Table of Contents Width

VM was reset (again). Re-cloned all 3 repos, re-installed TeXLive.

### Problem
VLM analysis of the TOC page revealed: "the right edge of the content is offset from the page's right edge by a wider margin (likely 2-3 cm)." The TOC content didn't span the full available width. Specific issues:
1. Fixed-length TikZ leader dots (3cm, 2.5cm, 2cm) that didn't stretch to the right margin
2. Extra `\hspace{22pt}` indent in `\maketoc` pushing content too far right
3. `\maketoc` used `remember picture, overlay` (same fragile approach as old chapter pages)
4. Duplicate "Contents" heading (from both `\renewcommand{\contentsname}` and `\maketoc`)

### Solution
1. **Replaced fixed TikZ leaders with `\dotfill`**: Overrode `\dotfill` globally with styled dotted leaders that stretch with `\hfill` to fill remaining space. Page numbers placed after the fill with `\hspace{0.5em}` gap.
2. **Rewrote `\maketoc`**: Removed `remember picture, overlay`. Used `\newgeometry` for the TOC page with custom margins. Simple "Contents" heading with decorative rule, no TikZ overlay needed.
3. **Fixed duplicate heading**: Set `\renewcommand{\contentsname}{}` before `\tableofcontents` to suppress the automatic TOC title (since `\maketoc` provides its own styled heading).
4. **Removed extra `\hspace{22pt}`**: TOC content now starts at the normal left margin.

### VLM Verification
- **Before**: Content confined to narrow column, short leader dots, significant empty space on right
- **After**: "dotted leader lines extend properly from the section titles to the page numbers" — **9/10**
- Content pages: no regressions

### Files changed
- `latex-template/sudodoc.sty` — v1.1.0 → v1.2.0, rewrote TOC section + `\maketoc`
- `latex-template/sample.pdf` — recompiled (9 pages)
- `latex-template/output/page-*.png` — updated previews

### Lessons
- Fixed-length TikZ leaders are fragile — use `\dotfill`/`\hfill` for flexible leaders that adapt to content width
- `\renewcommand{\contentsname}{}` must be set BEFORE `\tableofcontents` is called, and outside of a group
- `\newgeometry` works for TOC pages too — simpler than `remember picture, overlay`
- When testing TOC changes, always delete `.aux` and `.toc` files for a clean compile

## latex-8: Fix Dark/Light Mode Rendering Inconsistencies

VM was reset (again). Re-cloned all 3 repos (workspace, todo-list, latex-template), re-installed TeXLive (~25 min).

### Problem
The template had NO theme switching mechanism. Title page and chapter pages were always dark-themed (dark navy backgrounds), while body content was always light (white paper). There was no way to produce a light-themed document, and the lack of a semantic color system meant any future theme changes would require editing color definitions scattered throughout the entire style file.

### Solution: Full theme system in sudodoc.sty v1.3.0
Implemented a comprehensive dark/light theme system:

1. **Base palette**: Defined 13 shared color constants (cDarkNavy, cRoyalBlue, cDeepBlue, etc.) that never change — these are the raw building blocks.

2. **Semantic colors**: Created 28 semantic color aliases (cPageBg, cAccent, cTitleFg, cBodyFg, cTableHeadBg, etc.) that are remapped when the theme changes. All template elements reference ONLY semantic colors.

3. **Theme toggle**: Added `theme` option to `\sudodocsetup`:
   - `\sudodocsetup{theme=dark}` — default, dark title page + chapter pages
   - `\sudodocsetup{theme=light}` — white title page with blue accent band, blue chapter pages

4. **Theme-aware elements**: Title page, chapter pages, TOC, section headings, headers/footers, tables, lists, code listings, and blockquotes all use semantic colors.

5. **Backward compatibility**: Legacy color names (titleDark, titleAccent, etc.) are auto-aliased to semantic colors via `\sudodoc@setlegacy`, so external documents using old color names still compile.

6. **Implementation**: Uses `etoolbox`'s `\newtoggle`/`\iftoggle` for the dark mode flag, `\colorlet` for efficient color aliasing, and `\apptocmd` to hook legacy alias updates into theme change functions.

### Light theme design
- **Title page**: White background, dark blue left accent stripe, 3cm blue top decorative band, dark navy title text, gray subtitle/author
- **Chapter pages**: Blue (#0F3460) header band (instead of dark navy), blue accent stripe, white text
- **Body content**: Identical to dark theme (same table headers, code backgrounds, list colors)

### VLM Verification (side-by-side comparisons)
| Element | Dark | Light | Notes |
|---------|------|-------|-------|
| Title page | 7/10 | 7/10 | VLM had trouble reading title text in both (perception issue, not rendering) |
| TOC | 9/10 | 9/10 | Identical structure, both render correctly |
| Chapter pages | 7/10 | 8-9/10 | Light theme slightly preferred |
| Tables | 9/10 | 9/10 | Identical styling |
| Lists | 10/10 | 10/10 | Identical styling |
| Code listings | 10/10 | 10/10 | Identical syntax highlighting |
| Blockquotes | 10/10 | 10/10 | Identical border/background |

### Files changed
- `latex-template/sudodoc.sty` — v1.2.0 → v1.3.0, full theme system rewrite
- `latex-template/sample.tex` — unchanged (still uses default dark theme)
- `latex-template/sample-light.tex` — NEW, demonstrates light theme
- `latex-template/sample.pdf` — recompiled dark (9 pages)
- `latex-template/sample-light.pdf` — NEW, compiled light (9 pages)
- `latex-template/output/dark/` — dark theme PNGs
- `latex-template/output/light/` — light theme PNGs

### Lessons
- Use semantic color aliases from the start — makes theme switching trivial
- `\colorlet` is efficient for aliasing colors (no need for `\definecolor` with full specs)
- `\iftoggle` from etoolbox is cleaner than `\ifx\foo\bar` string comparisons
- Legacy color aliases via `\apptocmd` hooks ensure backward compatibility without code duplication
- Always use `Write` tool (not `Edit`) for todo.md to avoid formatting corruption

## latex-9: Redesign Section Headings with Colored Full-Width Boxes

No VM reset — all repos and TeXLive still present from previous run. Pulled latest.

### Problem
The current section headings used a simple TikZ underline + badge approach: bold dark text with a blue rule for `\section`, a blue number badge for `\subsection`, and a bullet for `\subsubsection`. Robby_Blue requested full-width colored boxes with white text, varying by heading level.

### Solution: Colored full-width boxes (sudodoc.sty v1.4.0)
Replaced all three heading styles:

1. **`\section`**: Full-width dark navy (`#1A1A2E`) `\colorbox` with bold white text, 14pt horizontal padding, 7pt vertical padding, subtle right-side decorative rule (40% opacity, 25% line width). Clean, prominent, readable.

2. **`\subsection`**: Full-width royal blue (`#0F3460`) `\colorbox` with white text and a darker number badge (`#1A6B8A` rounded corners), 6pt padding, subtle right-side decorative rule. Visually distinct from sections but clearly subordinate.

3. **`\subsubsection`**: Left accent bar (teal `#2A7B9B`) using a thin `\colorbox` rule with 10pt offset, followed by bold gray text. No full box — lighter treatment that avoids visual overload while maintaining hierarchy.

All three levels use the `\makebox[\linewidth][l]` + `\colorbox` + `\parbox` approach for reliable full-width rendering without TikZ overlays.

### Theme integration
- Added 5 new base colors (cSectionBox, cSubsectionBox, cSubsubAccent, cSectionBoxAccent, cSubsectionBoxAccent)
- Added 8 semantic color aliases (cSecBoxBg/Fg/Accent, cSubsecBoxBg/Fg/Accent, cSubsubAccent) to both dark and light theme functions
- Heading box colors are IDENTICAL in both themes (they're always on white body pages)

### VLM Verification

- Dark vs Light comparison: **10/10 both** — "colored heading boxes are identical"

### Files changed

### Lessons

## tts-research: Deep-Dive Research on German TTS Engines

VM was reset. Re-cloned all 3 repos (workspace, todo-list, latex-template). No TeXLive needed for this task.

### What was done
Comprehensive web research on 12 TTS engines for German language support on our hardware (4 vCPU, 8 GB RAM, no GPU). Used two parallel research agents to cover all engines simultaneously.

### Research findings
**12 engines investigated, 5 eliminated:**
- **MeloTTS** — No German support (Issue #145 open, no model)
- **StyleTTS2** — English only, author unresponsive
- **Fish Speech** — SOTA quality but needs 24 GB VRAM (way too heavy)
- **MMS TTS (Meta)** — Robotic quality + CC-BY-NC non-commercial license
- **Thorsten-Voice** — Not a separate engine (voice dataset used by Piper)

**7 viable engines ranked:**
1. **Piper TTS + Thorsten** — Best overall. Native German, trivially runs on our hardware (114 MB model), GPL-3.0 + CC0, excellent quality, single male speaker with 8 emotional variants
2. **Kokoro-82M German** — Community model (huggingFresse), 82M params, 1 male voice "Martin", permissive license, worth testing
3. **ZeroVOX** — Native EN+DE, ultra-lightweight FastSpeech2, pip install zerovox, but early alpha quality
4. **Bark (Suno)** — MIT license, expressive, 100+ presets, but hallucinations + 13s limit + tight on 8 GB
5. **CosyVoice (300M)** — Apache 2.0 (best license), LLM-based, voice cloning, but complex install + tight memory fit
6. **F5-TTS** — Excellent German fine-tunes, lightweight (~2-3 GB), but CC-BY-NC non-commercial weights
7. **Coqui XTTS v2** — Mature German support, voice cloning, but abandoned project + non-commercial license

### Key insight: No female German voice
None of the open-source engines that fit our hardware provide a female German voice out of the box. All are male-only. Female voice would require either Coqui XTTS v2 voice cloning (non-commercial) or a commercial API.
